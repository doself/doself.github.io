ART运行时的创建和初始化流程

发表于 2018-08-31 | 阅读次数:

启动Zygote

在 Android 系统中，所有的应用程序都是由 Zygote 进程 fork 出来的。Zygote 进程在启动时会创建一个 ART 虚拟机实例，每当它 fork 一个新的应用程序进程时，也会将这个 Art 虚拟机实例复制到新的应用程序进程里，从而使得每一个应用程序进程都拥有一个独立的 Art 虚拟机实例。
Zygote 程序的主函数，它的定义在 /frameworks/base/cmds/app_process/app_main.cpp 中。通过简单阅读 main 方法，可以发现所有的工作都被一个 AppRuntime 接管，是 AppRuntime 类的核心函数 start 启动了虚拟机。

main方法中省略的代码主要在处理传入的参数, 从而确定当前是否要执行 zygote 的逻辑以及启动 zygote 时是否需要启动 system_server，这里主要两个关键调用：

  • AppRuntime runtime(...)这行代码创建了一个 AppRuntime 对象，AppRuntime 继承自 AndroidRuntime
  • runtime.start(...)调用 AppRuntime 的 start 方法，由于AppRuntime并没有重写父类的start方法,所以这里实际调用的是 AndroidRuntime::start(...)

Android运行时的创建与初始化

AndroidRuntime::start(...)方法的作用是启动Android Runtime，主要涉及到两方面的操作：

  1. 启动虚拟机，包括：
    • 创建一个 JniInvocation 实例，并且调用它的成员函数 init 来初始化 JNI 环境
    • 调用 startVm 方法，创建一个 ART 虚拟机实例
    • 调用 startReg 方法注册 JNI 函数表
  2. 执行”com.android.internal.os.ZygoteInit”或”com.android.internal.os.RuntimeInit”类下的main方法，启动孵化器 ZygoteInit 或 RuntimeInit

启动ART虚拟机

新建一个JniInvocation实例，并初始化相应的JNI函数接口

JniInvocation 中有三个重要的函数指针：

  • JNICreateJavaVM（创建虚拟机实例）
  • JNIGetCreatedJavaVMs（获取创建的虚拟机实例）
  • JNIGetDefaultJavaVMInitArgs（获取虚拟机的默认初始化参数）

这三个变量是标准的 Java 虚拟机的接口，也就是说实现了这个三个接口，就可以实现一个 Java 虚拟机。

在 init 初始化方法中加载 libart.so ，寻址”JNI_CreateJavaVM”、”JNI_GetCreatedJavaVMs”、”JNI_GetDefaultJavaVMInitArgs”这些符号，并将获取到的符号地址赋给以上函数指针。

创建ART虚拟机

AndroidRuntime::startVm 主要做了2件事：

  1. 解析虚拟机启动参数
  2. 调用JNI_CreateJavaVM创建虚拟机

这里可以看到实际是调用了 JNICreateJavaVM 方法，而 JNICreateJavaVM 就是之前提及过的在 JniInvocation::Init 中导出的函数指针。

然而我们还是想看下 JNICreateJavaVM 函数的具体实现，实际上 libart.so 的代码都位于 /art 目录下，而 JNI_CreateJavaVM 则是定义在路径 /art/runtime/java_vm_ext.cc 中。

具体代码实现

JNICreateJavaVM 函数代码主要完成以下几件工作：

  • 调用 Runtime::Create 方法，在进程中创建一个 ART 虚拟机
  • 创建成功后，调用 Runtime 类的另外一个静态成员函数 Start 启动该 ART 虚拟机
  • 获取 JniEnv 和 JavaVM 指针，返回给上一层，以确保创建虚拟机后每个进程都有一个 JavaVM 指针，且虚拟机中的每个线程都有自己独立的 JNI 环境

初始化ART虚拟机运行时环境

代码位于 /art/runtime/runtime.cc ，可以看到该函数首先会判断当前进程是否已经创建有一个ART虚拟机实例。如果有的话，就立即返回。否则的话，就创建一个ART虚拟机实例，并且保存在 Runtime 类的静态成员变量 instance_ 中，最后调用 Runtime 类的成员函数 Init 对该新创建的 ART 虚拟机进行初始化。

Runtime:Init方法代码量较大，总结起来，主要有以下几个方面：

  • 子模块的初始化，如 MemMap::Init 初始化内存映射模块等，这部分没有太多值得关注的
  • 对传入的参数进行解析，初始化设置基础变量，如：
    • boot_class_pathstring
    • class_pathstring
    • patchoatexecutable
    • threadlist
    • interntable
    • …
  • 新建一个OatFileManager实例，便于后续加载OAT文件等
  • 创建ART虚拟机堆，ART虚拟机堆使用一个Heap对象来描述
  • 调用InitPlatformSignalHandlers()方法，实现自己的信号处理函数，对于NullPointerException、StackOverflowExeption等运行时异常，ART都是通过linux信号机制实现的。例如，ART会截取SIGSEGV信号，如果发现这是一个空指针问题，且发生空指针的代码处于java代码中，它就会产生一个NullPointerException抛出，然后继续执行
  • 创建一个JavaVMExt实例，里面的成员变量functions保存了虚拟机实例的相关函数表，使得我们可以将当前线程Attach到ART虚拟机中，又或者销毁当前进程的ART虚拟机等
  • 将当前线程Attach到ART虚拟机的主线程，使其可以调用ART虚拟机所提供的JNI接口。有了这个JNI环境之后，我们才能够在Java函数和C/C++函数之间互相调用
  • 获取当前的ART虚拟机堆，通过

Runtime类的成员函数GetHeap返回的便是当前ART虚拟机的堆，也就是前面创建的ART虚拟机堆。通过调用Heap类的成员函数GetContinuousSpaces可以获得堆里面的连续空间列表。如果这个列表的第一个连续空间是一个Image空间，那么就调用ClassLinker类的静态成员函数CreateFromImage来创建一个ClassLinker对象。否则的话，上述ClassLinker对象就要通过ClassLinker类的另外一个静态成员函数CreateFromCompiler来创建。创建出来的ClassLinker对象是后面ART虚拟机加载加载Java类时要用到的。
后面我们分析ART虚拟机的垃圾收集机制时会看到，ART虚拟机的堆包含有三个连续空间和一个不连续空间。三个连续空间分别用来分配不同的对象。当第一个连续空间不是Image空间时，就表明当前进程不是Zygote进程，而是安装应用程序时启动的一个dex2oat进程。安装应用程序时启动的dex2oat进程也会在内部创建一个ART虚拟机，不过这个ART虚拟机是用来将DEX字节码编译成本地机器指令的，而Zygote进程创建的ART虚拟机是用来运行应用程序的。
Runtime类的成员函数Init首先调用ParsedOptions类的静态成员函数Create对ART虚拟机的启动参数raw_options进行解析。解析后得到的参数保存在一个ParsedOptions对象中，接下来就根据这些参数一个ART虚拟机堆。ART虚拟机堆使用一个Heap对象来描述。
